# CulturalBridge 使用手册设计文档

## 1. 背景与目标

CulturalBridge（MCA Translation）功能模块日趋完整，涵盖文化感知翻译、风险标注、审校服务、术语库、历史记录、管理员后台等。为帮助内容编辑/译者和管理员快速上手并深入理解各功能，需编写一份统一的使用手册，并同时以仓库文档和应用内帮助页面两种形式交付。

**目标：**
- 降低新用户学习成本，覆盖核心工作流与常见问题。
- 为管理员提供独立的操作指引。
- 保持仓库文档与应用内帮助内容一致，减少维护负担。

## 2. 目标读者

手册按角色分章节，同时服务两类读者：

- **内容编辑 / 译者**：使用翻译工作台、风险标注、审校、历史记录、术语库等功能。
- **系统管理员**：管理用户、调整积分、查看登录状态、创建初始管理员。

## 3. 交付物

1. **仓库文档**：`docs/USER_MANUAL.md`，标准 Markdown 格式，纳入版本控制。
2. **应用内帮助页面**：前端新增 `/help` 路由，渲染同一份 Markdown 内容。
3. **配图资源**：`docs/help/` 目录下的截图与示意图（构建时同步到 `frontend/public/help/`）。

## 4. 内容结构

### 第 1 章 快速开始
- 登录与注册
- 界面概览（导航、主要页面）
- 完成第一次翻译

### 第 2 章 内容编辑 / 译者指南
- **工作台翻译**
  - 输入文本与文件上传（.txt / .docx / .pdf）
  - 选择文体、目标语言、文化圈、受众类型
  - 启动翻译与查看结果
- **理解风险标注**
  - 风险类型与含义
  - 接受 / 忽略 / 回退操作
- **高语境术语内联高亮**
  - 术语库命中与 LLM 识别的文化负载词
  - 悬停查看分类、风险备注、转译建议
- **转译决策日志**
  - 各阶段决策说明
  - 如何结合日志理解译文
- **历史记录**
  - 筛选、查看详情
  - 加载到工作台 / 删除
- **审校服务**
  - 对照审校 vs 独立审校
  - 四维评分解读
  - 内联标注与问题卡片
  - 导出 Markdown 审校报告
- **导出译文**
  - .txt 纯文本导出
  - .docx 原文 + 译文 + 风险批注导出
- **术语库管理**
  - 查看系统知识库
  - 添加 / 编辑 / 删除自定义术语
  - 一键补齐多语言译法

### 第 3 章 管理员指南
- 初始管理员创建
- 用户管理（启用 / 禁用 / 删除）
- 积分调整
- 查看登录状态

### 第 4 章 常见问题（FAQ）
- 翻译结果不理想怎么办
- 风险标注误报如何处理
- 术语库未命中怎么办
- 审校评分怎么看
- 账号、积分、登录相关问题

### 第 5 章 场景教程
- 场景 1：政治话语对外传播翻译
- 场景 2：文化隐喻的目标语适配
- 场景 3：多轮审校与导出工作流

## 5. 技术方案

### 5.1 文件结构

```
mca-translation/
├── docs/
│   ├── USER_MANUAL.md          # 手册唯一源文件（中文）
│   └── help/                   # 手册截图与示意图源文件
│       ├── workspace-overview.png
│       ├── risk-annotation.png
│       ├── decision-log.png
│       ├── history-detail.png
│       ├── review-scoring.png
│       ├── glossary-page.png
│       └── admin-users.png
├── frontend/
│   ├── app/(main)/help/
│   │   ├── page.tsx            # /help 帮助中心页
│   │   └── layout.tsx          # 帮助页布局
│   ├── components/help/
│   │   └── help-content.tsx    # Markdown 渲染组件
│   ├── lib/help.ts             # 加载/解析 Markdown 工具
│   └── public/help/            # 构建产物：docs/help/ 的同步副本
```

### 5.2 Markdown 渲染

- 使用 `react-markdown`（或兼容 Next.js 16 的同类库）解析 `docs/USER_MANUAL.md`。
- 支持 GitHub Flavored Markdown：表格、代码块、引用、锚点链接。
- 根据 `##` / `###` 标题自动生成目录，支持点击跳转。

### 5.3 图片路径处理

- 源图片存放在 `docs/help/`。
- Markdown 源文件中写作 `![说明](./help/xxx.png)`，保证在 GitHub 仓库中可预览。
- 构建时通过脚本将 `docs/help/` 同步到 `frontend/public/help/`。
- 应用内渲染时，将路径前缀 `./help/` 替换为 `/help/`，使 Next.js 正确加载静态资源。

### 5.4 页面布局

- 顶部标题栏 + “返回工作台”按钮。
- 左侧固定目录导航。
- 中间正文区域，宽度适中，便于阅读。
- 右下角“回到顶部”按钮。

## 7. 应用内入口

- 在主布局顶部导航或用户下拉菜单中增加 **“使用手册”** 链接，指向 `/help`。
- 可选：在工作台、审校、术语库等页面标题旁加 `?` 图标，锚定到对应章节。

## 8. 维护机制

- `docs/USER_MANUAL.md` 是唯一事实来源。
- 新增或修改功能时，同步更新对应章节。
- UI 改版时，批量重新截取 `docs/help/` 中的截图，并同步到 `frontend/public/help/`。
- 建议把“是否更新使用手册”作为功能 PR 的自检项。

### 更新流程

```
功能变更 / UI 调整
  → 修改 docs/USER_MANUAL.md
  → 如有必要，替换 docs/help/ 截图并同步到 frontend/public/help/
  → 本地启动前端，访问 /help 验证渲染正常
  → 提交 PR
```

## 9. 验收标准

- [ ] `docs/USER_MANUAL.md` 包含完整的中文手册内容，结构清晰。
- [ ] `/help` 页面可正常访问并渲染 Markdown 内容。
- [ ] 目录导航可点击跳转，锚点链接正常工作。
- [ ] 图片在仓库预览和应用内均能正确显示。
- [ ] 应用内有明确的“使用手册”入口。
- [ ] FAQ 和场景教程覆盖用户常见痛点。
