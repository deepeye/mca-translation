# PRD：面向国际传播的文化适配 AI 转译系统
**版本 v1.0 | 2025**

---

# 第一章：产品概述

**CulturalBridge** 是一款面向国际传播的文化适配转译平台，让政府机构、国际媒体和智库的内容团队能够将中文政治话语、文化表达与叙事逻辑精准适配到 30+ 种语言的目标受众认知框架，而无需依赖既不懂中文语境又不懂目标文化的传统翻译外包链条。

---

## 1.1 差异化对比表

| 功能维度 | DeepL / Google Translate | 人工翻译公司 | **CulturalBridge** | 实现方式 |
|---------|--------------------------|-------------|-------------------|---------|
| 政治话语适配 | 直译，语义漂移严重 | 依赖译者个人理解，质量不稳定 | 基于话语体系映射的语义等值转译 | 构建政治话语-目标语言认知框架对照知识库，LLM 动态检索适配 |
| 叙事逻辑重构 | 不处理 | 人工判断，无系统标准 | 自动识别叙事结构并按目标受众思维习惯重排 | 叙事结构解析器 + 文化逻辑模板引擎 |
| 文化隐喻转换 | 字面翻译 | 取决于译者跨文化能力 | 隐喻等效替换，保留情感强度 | 中文文化隐喻库 × 目标语言等效隐喻匹配 |
| 传播效果预测 | 无 | 无 | 输出前预测受众接受度评分 | 基于目标语言媒体语料的接受度模型 |
| 多语言批量处理 | 支持但无文化适配 | 慢（数天交付） | 30+ 语言并行，单文档 < 3 分钟 | 异步并行 API 调用 + 结果合并 |
| 版本追踪与溯源 | 无 | 无 | 每次转译完整记录决策链 | 转译决策日志 + diff 可视化 |

## 1.2 三类用户画像

**画像 A：国际传播内容编辑（机构媒体）**
- 角色：在国际广播机构或官方媒体负责多语言内容发布的编辑
- 核心目标：在截止日期前产出在目标市场真正可读、有说服力的内容
- 对现有工具最大不满：机器翻译输出像新闻稿翻译腔，人工翻译周期太长，两者都无法处理「五位一体」「新型举国体制」这类高语境词汇
- 让他切换的那一个功能：**政治话语转译建议**——系统给出 3 种不同适配方案并标注各自的受众定位风险

**画像 B：智库研究员（政策倡导方向）**
- 角色：为国际受众撰写中国政策解读报告的研究员
- 核心目标：让目标读者（欧洲政策官员、东南亚学者）真正理解政策意图而非产生误读
- 对现有工具最大不满：没有工具能告诉他「这个表达在目标受众那里会触发什么认知定势」
- 让他切换的那一个功能：**受众认知风险标注**——系统在输出文本中标注哪些词汇/表达在目标文化中存在负面联想风险

**画像 C：跨国企业本地化负责人**
- 角色：中国总部品牌或公关内容的国际化团队负责人
- 核心目标：确保品牌叙事在跨文化传播中保持一致性，不在某个市场引发公关危机
- 对现有工具最大不满：本地化团队各自为政，同一段话在不同市场的译文传递了截然不同的信息
- 让他切换的那一个功能：**跨语言叙事一致性审计**——上传多语言版本，系统自动检测核心信息漂移

## 1.3 可行性边界

| 在范围内（及原因） | 明确排除在外（及原因） |
|-----------------|-----------------|
| 中文 → 30+ 主流语言文字转译（LLM 能力覆盖） | 实时语音同传（延迟要求 < 500ms 与文化适配深度不可兼得） |
| 政治话语、新闻报道、政策文件、品牌传播四类文体 | 文学翻译（审美判断无法标准化，不在传播场景需求内） |
| 文化适配建议与风险标注 | 法律文件认证翻译（需持证资质，产品不承担法律责任） |
| 批量文档处理（PDF、Word、纯文本） | 图片/视频内嵌文字识别转译（MVP 阶段排除，OCR 管线独立） |
| 转译决策可解释性（用户可查看为什么这样翻） | 自动发布到第三方平台（仅输出，不代替用户的发布决策） |
| 用户自定义术语表和风格指南 | 承诺任何语言对的转译质量均等（低资源语言质量会显著下降，需明确标注） |

**布局逻辑说明：**

**b) 交互流程**

```
用户粘贴/上传文本
        |
        v
自动文体检测（< 500ms）
        |
        +-- 检测置信度 > 85% --> 显示「已自动检测为：XX 文体」+ 允许手动修正
        |
        +-- 检测置信度 < 85% --> 提示「请确认文体」+ 高亮候选项
        |
        v
高语境术语识别（内联标注）
        |
        +-- 发现政治话语术语 --> 蓝色下划线 + 悬停显示转译建议预览
        |
        +-- 发现文化隐喻     --> 橙色下划线 + 标注「存在文化等效替换方案」
        |
        v
用户点击「开始转译」
        |
        v
── 失败路径 A：文本为空 ──────────────────────────
输入框边框变红 + 提示「请先输入或上传文本」
        |
── 失败路径 B：目标语言未选择 ─────────────────────
右侧语言选择器高亮闪烁 + 提示「请至少选择一种目标语言」
```

**c) 状态清单**

| 状态名 | 触发条件 | 视觉标识 | 退出条件 |
|--------|---------|---------|---------|
| 空态 | 初始进入 / 清空后 | 灰色占位文字 + 示例按钮 | 用户开始输入 |
| 输入中 | 用户正在输入 | 字数计数器实时更新 | 用户停止输入 > 800ms |
| 分析中 | 停止输入 800ms 后 | 术语下划线逐步显示 | 分析完成 |
| 已就绪 | 分析完成且已选目标语言 | 「开始转译」按钮激活 | 用户点击转译 |
| 错误-文件格式 | 上传不支持的格式 | 红色提示条 | 用户关闭提示 |
| 错误-超限 | 文件 > 10MB | 红色提示条 + 处理建议 | 用户重新选择 |

**d) 依赖关系**

- **读取自**：术语库模块（用户自定义术语）、风格指南模块
- **写入至**：转译引擎模块（文本内容 + 文体标签 + 术语标注）、决策日志模块

**e) 待决问题**

1. 自动文体检测的模型：使用专门微调的分类器还是让主 LLM 兼做分类？前者更快更准，但需要维护额外模型；后者实现简单，但延迟增加约 300ms。
2. 超过 3000 字的长文档是否分段处理？分段会损失段落间的叙事连贯性，但单次处理长文档会导致 token 成本急剧上升。

**d) 依赖关系**

- **读取自**：输入模块（文本+标注）、术语库模块、风格指南模块、文化知识库（内部）
- **写入至**：决策日志模块、历史记录模块、接受度评分模块

**e) 待决问题**

1. 「受众优先」策略下的叙事重排边界：系统有权改变段落顺序，但是否允许删减论点？删减可能被视为内容审查。当前倾向：只调序不删减，用[…]标注建议删减的段落。
2. 接受度评分的语料基准：使用目标语言主流媒体语料还是社交媒体语料？两者结果差距可能达 20 分以上。

# 第四章：超越竞品的差异化功能

## 4.1 政治话语语义等值转译

**第一段 — 竞品为何没有这个功能：**
传统机器翻译公司（DeepL、Google）的架构基于统计语料对齐，其训练数据中中文政治话语对应的英文语料大多来自官方英译版本（如新华社英文电稿），而这些英译本本身存在语义漂移——它们已经是「对外宣传英语」，而非目标受众习惯的英语表达。系统在学习时把这个漂移复制进模型，形成「双重漂移」。专业翻译公司的问题是不同，他们的译者虽然理解目标语言，但极少有人同时深谙中国政治话语体系的内部逻辑，导致质量完全依赖个人，无法标准化。

**第二段 — 本产品如何实现：**
我们构建了「话语意图层」分离机制：转译流程的第一步不是翻译词汇，而是识别该话语在中文政治语境中的**功能定位**（是宣示性的、解释性的、动员性的还是外交性的），然后根据功能定位选择目标语言中能实现同等功能的话语策略。例如「新型举国体制」在宣示场景下对英美政策受众应展开为「state-coordinated national mobilization system」（让受众明白机制），在新闻场景下可简化为「China's centralized innovation model」（让受众感知差异）。技术实现上，我们用结构化 prompt + RAG（检索话语知识库）而非端到端生成，原因是端到端生成在低频政治话语上幻觉率高达 40%。

**第三段 — 交互流程：**

```
用户输入包含「新型举国体制」的政策文件
        |
        v
文体识别 → 政策文件 + 目标受众：英美政策媒体
        |
        v
话语意图识别 → 「宣示性 + 解释性」
        |
        v
检索话语知识库 → 找到该术语的 3 种功能场景译法
        |
        v
输出：「state-coordinated national mobilization system」
+ 决策说明：「在政策媒体受众中，解释性译法接受度比直译高 34%」
+ 可选替换：[点击查看其他 2 种方案]
        |
        v
用户可在结果中直接点击术语 → 弹出译法比较器 → 选择并锁定该译法
```

**第四段 — 风险与应对：**
主要风险：话语知识库的覆盖率和更新频率。政治话语随党代会、重要讲话、政策周期实时演化，若知识库滞后 6 个月，转译建议可能出现「话语代差」。应对方案：（1）建立术语「新鲜度」标签，超过 12 个月未更新的条目降权并提示用户；（2）当系统遇到知识库未收录的新术语时，不静默处理而是在输出中明确标注「⚠ 此术语为新出现术语，建议人工审校」；（3）允许用户在术语库中提交新发现的术语，由运营团队定期审核纳入。

## 4.2 受众认知风险预测

**第一段 — 竞品为何没有这个功能：**
翻译工具的产品逻辑是「语言 A → 语言 B 的保真度」，不涉及「语言 B 的内容在目标受众中产生什么效果」。这个缺口的根源在于商业模式：翻译公司收费依据是词数或语对，接受度预测需要对目标文化有实质性建模能力，且这个能力很难标准化定价。此外，「我们帮你翻了，翻完传播效果如何与我无关」是整个行业的隐性免责声明。

**第二段 — 本产品如何实现：**
我们用目标语言主流媒体语料（分政治立场）训练了一个轻量级接受度分类器（BERT-based，按语言分别训练），输入转译后的文本，输出 0-100 的接受度分数和「风险词/表达」列表。分类器不评判内容的价值观，只预测「目标受众是否会以预期方式理解这段内容」。选择 BERT-based 分类器而非让主 LLM 做评估的原因是：LLM 在这类评估任务上存在政治立场漂移，其评估结果与真实受众反应的相关性难以验证；专门训练的分类器虽然上限低，但在已知域内的一致性高，且可以独立审计。

**第三段 — 交互流程：**

```
转译完成
        |
        v
接受度分类器对转译文本打分（< 2s）
        |
        v
生成风险标注：
文本中的风险词汇/短语获得内联高亮
每个高亮点 hover 显示：
  ├─ 风险等级：低/中/高
  ├─ 风险类型：认知偏差 / 情感负联想 / 理解歧义
  └─ 建议：[保持原译] [查看替代方案]
        |
        v
右侧面板显示整体评分 + 最高风险条目前 3 位
        |
        v
用户点击某个风险高亮 → 跳转至转译替代方案 → 可直接替换
        |
        v
替换后实时重新计算评分（delta 更新，< 1s）
```

**第四段 — 风险与应对：**
最大风险是分类器的偏见：用主流媒体语料训练意味着分类器的「接受度」基准是主流媒体受众，而非全部目标受众。这可能导致系统对非主流但合理的表达方式打低分。应对：（1）允许用户选择「受众基准」（主流媒体 / 学术界 / 社交媒体），切换不同的分类器权重；（2）评分界面明确标注「基于[XX受众]的接受度预测，仅供参考」；（3）接受度评分是建议，不是限制，用户可以忽略。

## 4.3 跨语言叙事一致性审计

**第一段 — 竞品为何没有这个功能：**
传统翻译工具的设计假设是「每次翻译相互独立」，没有「同一文档的多语言版本应保持核心信息一致」的产品概念。本地化公司虽然会做内部 QA，但依赖的是双语译者逐对比对，不能覆盖 10+ 语言，成本也无法承受。这是一个被行业结构性忽视的需求：用翻译工具的人和关心品牌一致性的人，通常是不同的团队。

**第二段 — 本产品如何实现：**
我们提取每个语言版本的「核心命题集」（claim extraction），将其转为语言无关的语义向量，然后计算各语言版本的命题集合差异：哪些核心命题在某语言版本中被删减？哪些被弱化？哪些被强化？哪些被添加？使用 multilingual embedding 模型（如 LaBSE），将跨语言语义对比转化为向量距离计算。

**第三段 — 交互流程：**

```
用户进入「一致性审计」功能
        |
        v
上传同一文档的多语言版本（或选择历史记录中的转译结果）
        |
        v
系统提取各语言版本的核心命题（约 30s/文档）
        |
        v
生成一致性矩阵：
  命题 × 语言 的表格
  ├─ ✓ 完整保留
  ├─ △ 部分保留（强度降低）
  ├─ ⚠ 添加（原文无）
  └─ ✗ 缺失
        |
        v
用户点击某个「⚠ 缺失」单元格
→ 显示：「该命题在德语版本中未找到对应表达」
→ 显示：原文段落 + 德语版本对应段落 + 建议补充译文
        |
        v
用户确认修改 → 更新对应语言版本 → 重新计算一致性分数
```

**第四段 — 风险与应对：**
命题提取的精度是核心风险：LLM 在提取「核心命题」时可能将修辞性表达误识别为实质性命题，导致误报；也可能漏掉隐含的重要命题。应对：（1）命题提取结果可展示给用户审核，不直接用于生产决策；（2）提供命题置信度标注，低置信度命题不计入一致性评分；（3）将此功能定位为「辅助审计工具」而非「自动合规检查」，在 UI 上明确传达。

| ONNX Runtime | 接受度分类器推理 | 将训练好的分类器导出为 ONNX 格式，推理延迟比原生 PyTorch 低 40-60%；无需 GPU 即可达到目标延迟 | 未知（服务端） |
| Zustand | 前端状态管理 | 比 Redux 轻量（包体积小 ~10x），API 简洁，对转译任务状态机（多状态、多语言结果）的建模比 React Context 清晰 | ~2.5KB gzip |
| react-diff-viewer | 原文/译文对比展示 | 专为 diff 显示优化，比自实现成本低；支持内联模式和分栏模式，适合转译对比场景 | ~15KB gzip |

**最大架构风险：**
LLM API 的可用性和延迟是最大风险。单次政治话语转译调用链包含多个 LLM 步骤（文体识别、术语处理、主转译、接受度评估），总延迟可达 20-40 秒，任何一步超时都会导致用户体验断裂。应对策略：（1）使用流式输出（streaming）在第一个 token 产出时即开始向用户展示，心理等待时间大幅降低；（2）文体识别和术语识别使用更小、更快的专用模型，不调用主 LLM；（3）为 Anthropic API 设置备用方案（fallback 到其他 provider），但需评估话语转译质量差异；（4）对高频重复文本（例如固定术语）实现结果缓存，避免重复调用。

## 8.3 批量处理流程

```
用户提交批量任务（N 个文件 × M 种语言）
        |
        v
格式校验（并行，< 5s）
        |
        +-- 通过文件 → 进入处理队列
        +-- 失败文件 → 标记跳过，记录原因，不阻断
        |
        v
队列调度（按文件大小升序）
        |
        v
┌─────────────────────────────────────────┐
│  文件 1 ──────────────────────────────  │
│          ├─ 英语转译 ←─ 并行 ─→ 德语转译 │
│          └─ 日语转译 ←─ 并行 ─→ 法语转译 │  每个文件的多语言并行处理
│  文件 2 ──────────────────────────────  │
│  文件 3 ──────────────────────────────  │

        |
        v
结果聚合
        |
        +-- 全部成功 → 生成 ZIP
        +-- 部分失败 → ZIP 含成功部分 + failures.json（含错误原因）
        |
        v
若处理时间 > 2 分钟 → 发送邮件通知 + 提供下载链接（有效期 7 天）
```

# 第九章：开发优先级

**P0 — 没有这个，产品根本无法使用。**
交付标准：功能可用，不需要完美。

- 文本输入界面（粘贴 + 文件上传）
- 文体手动选择（自动检测可降级为占位符）
- 目标语言选择（英语、德语、日语、西班牙语、法语先支持 5 种）
- 调用 LLM 进行基础转译（无知识库增强，直接 prompt）
- 结果展示与复制
- 单语言导出（.txt）

**P1 — 没有这个，用户第一次体验后不会回来。**
交付标准：功能完整，体验流畅。

- 政治话语知识库接入（RAG 增强转译）
- 高语境术语内联高亮与悬停预览
- 转译决策日志（基础版，文字描述）
- 风险标注（基础版，仅标注位置，不给出建议替换）
- 多语言并行处理（最多 5 种同时）
- .docx 导出
- 用户自定义术语表（创建 + 在转译中应用）
- 历史记录查看

**P2 — 有了这个，用户会把产品推荐给别人。**
交付标准：稳定且有辨识度。

- 接受度评分（英语、德语、日语先支持）
- 风险标注完整版（含替换建议）
- 叙事重排（「受众优先」策略完整实现）
- 批量处理（最多 20 个文件）
- 一致性审计（两个语言版本对比）
- 决策日志 PDF 导出
- 支持语言扩展至 15 种

**P3 — 有了这个，一部分用户会付费或强烈倡导。**
交付标准：精致且有完整文档。

- 跨语言一致性审计（10+ 语言）
- 接受度评分受众基准选择
- 组织级术语表共享
- API 接入（程序化调用）
- 支持语言扩展至 30+ 种
- 白标/私有化部署方案
- 转译质量趋势分析（随时间追踪同类文本的转译效果）

# 第十章：性能指标

| 指标名称 | 目标值 | 测量方法 | 劣化阈值 |
|---------|--------|---------|---------|
| 单文档转译完成时间（< 500字，英语） | p50 < 15s, p95 < 30s | 生产环境任务从提交到结果可见的端到端时长 | p50 > 25s（用户开始感知「太慢」） |
| 单文档转译完成时间（< 2000字，英语） | p50 < 35s, p95 < 60s | 同上 | p50 > 50s |
| 文体识别延迟 | p95 < 800ms | 从文本输入停止到术语高亮出现的时长 | > 1500ms（打断用户阅读节奏） |
| 接受度评分计算延迟 | p95 < 2s | 从转译完成到评分显示的时长 | > 3s（用户预期是转译完即有分数） |
| 知识库术语检索延迟 | p99 < 200ms | 向量相似度检索单次耗时 | > 500ms（影响批量任务的总时长） |
| 批量处理吞吐量 | 20 文件 × 5 语言 < 15 分钟 | 整批任务从提交到全部完成的时长 | > 30 分钟（用户会离开等待） |
| 前端首屏可交互时间 | < 2.5s（FCP < 1.5s） | Lighthouse 测量，冷启动，标准网络 | FCP > 3s |
| 前端 JS bundle 体积 | < 350KB gzip | webpack-bundle-analyzer 测量 | > 500KB（移动端加载体验明显下降） |
| API 可用性 | > 99.5%（月度） | 外部 uptime 监控（每 30s 探测） | < 99%（开始影响专业用户的工作流程依赖） |
| 低资源语言转译质量置信度 | 系统内置分类器评分 > 0.65 时才不显示警告 | 每次转译后分类器输出 | 不适用（低于阈值即显示警告，无劣化概念） |

# 第十一章：开发者交接说明

本章写给将要实现这个产品的 AI 编程助手。

## a) 实现顺序建议

**第一步：搭建转译核心管线**（解锁一切）
先实现最简单的端到端流程：用户输入文本 → 调用 Claude API → 显示结果。不加知识库、不加知识增强，只是通路。这一步让你理解 prompt 设计的基本模式，也能让 P0 功能快速可测试。

**第二步：实现知识库 + RAG 接入**（解锁差异化）
知识库是整个产品的核心资产，RAG 的检索逻辑质量直接决定政治话语转译的准确性。在正式实现前，用固定的 10-20 条术语做硬编码测试，验证 prompt 中术语注入的格式和效果，再开发通用的 RAG 检索机制。

**第三步：实现异步任务队列**（解锁批量和稳定性）
长时转译任务必须异步处理，否则任何网络抖动都会导致任务丢失。Celery + Redis 的搭建建议在第三步完成，不要等到做批量处理时才引入，因为单文档的 30-60 秒转译本身也需要异步机制。

**第四步：接受度评分和风险标注**（独立模块，可并行）
ONNX 推理服务可以作为独立微服务，与转译服务松耦合。接入方式是转译完成后调用评分 API，结果回填到 TranslationJob。这个模块可以独立开发和测试。

## b) 最可能导致返工的三个决策

**决策一：Prompt 结构设计**
决策内容：是否将文体识别、术语处理、主转译、叙事重排做为一次 LLM 调用，还是分多次调用？
安全默认选择：**分多次调用**。每步独立，错误定位更容易，输出质量更可控。合并调用的成本节省不值得承受联调的复杂度。
方向改变信号：如果延迟在分步调用下仍 > 45s（p50），再考虑合并前两步（识别+处理）。

**决策二：向量数据库 vs pgvector**
决策内容：知识库规模超过多少条时需要从 pgvector 迁移到专用向量数据库？
安全默认选择：**从 pgvector 开始**。当前设计的知识库上限约 10 万条，pgvector 完全可以胜任，且省去引入新服务的运维成本。
方向改变信号：检索 p99 超过 500ms，或知识库条目数超过 50 万时，评估 Qdrant 或 Weaviate。

**决策三：前端状态管理的粒度**
决策内容：多语言转译结果是否共享一个状态，还是每个语言独立状态机？
安全默认选择：**每个语言独立状态机**（Zustand slice per language）。共享状态在多语言并发更新时容易产生竞态问题，调试困难。独立状态机更清晰，也方便实现「单独重新转译某个语言」功能。
方向改变信号：如果用户需要「一次性对比全部语言的接受度评分」，独立状态需要合并读取逻辑，此时重构为共享状态对象（含各语言子状态）。

## c) 哪里要严格，哪里可以灵活

| 章节 | 类型 | 说明 |
|------|------|------|
| 第五章 数据模型 | **约束** | 字段名和结构不得随意更改，影响决策日志的溯源和导出格式 |
| 第十章 性能指标 | **约束** | 单文档 p50 < 15s 是产品可用的底线，不得通过降低转译质量换取速度 |
| 第七章 错误状态的提示文案 | **约束** | 错误提示必须包含「发生了什么」和「怎么处理」，不能只显示错误码 |
| 第三章 模块 UI 布局 | **建议** | ASCII 图展示的是信息层级，具体像素尺寸和视觉风格由你判断 |
| 第六章 依赖库 | **建议** | 表格中的库是推荐选择，如果你有更熟悉的等效库，可以替换，但需要验证包体积和性能不劣化 |
| 第九章 P0 功能集 | **约束** | P0 列表是最小可测试集合，不得在 P0 阶段引入 P1 功能 |
| 第四章 差异化功能的技术实现 | **建议** | 实现方案的技术路径由你判断，但功能的用户可见行为（输入→输出）是约束 |

## d) 已知的未知项

**未解决项 1：低资源语言的质量下限**
本 PRD 承诺支持 30+ 种语言，但系统知识库（政治话语术语、文化隐喻库）对低资源语言的覆盖率是未知的。实现者在支持每种语言时，需要实际测试知识库命中率，若 < 30%，该语言的转译实际上退化为纯 LLM 输出，「文化适配」的核心价值无法体现。建议：对知识库覆盖率 < 30% 的语言，在界面上明确标注「基础翻译模式」，不宣称文化适配能力。

**未解决项 2：「叙事重排」的用户授权边界**
「受众优先」策略下系统会重排段落，但 PRD 未明确定义「重排」的边界：是只调整段落顺序，还是允许合并/分割段落？在政治话语场景中，这个边界涉及内容授权问题（用户是否授权系统修改文档结构）。建议：在「受众优先」策略的首次使用时，显示一次明确的操作说明和用户确认弹窗，明确告知会做什么、不会做什么。

**未解决项 3：多租户/组织级数据隔离**
PRD 的数据模型包含 `owner_id` 字段支持组织级术语表，但组织内的权限管理（谁可以编辑共享术语表、谁只能读取）完全未设计。实现者遇到这个需求时，不要自行设计权限系统，而是暂时用「组织内所有成员可读写」的最简模型，等产品验证阶段结束后再设计完整的 RBAC。
